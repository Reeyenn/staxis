# Audit findings — 35 clusters from 51 verified findings

## 1. [CRITICAL] Team-management capabilities (manage_team / manage_users) default-granted to every hotel role — fleet-wide privilege escalation out of the box
- **rec:** needs-reeyen-decision  | **cat:** Auth/authorization bypass
- **files:** src/lib/capabilities/registry.ts, src/app/api/auth/team/route.ts, src/app/api/auth/invites/route.ts, src/app/api/auth/join-codes/route.ts, src/app/api/settings/users/route.ts, src/lib/capabilities/can.ts
- **root cause:** ROLE_DEFAULTS maps every non-adminOnly capability to DEFAULT_GRANTED_ROLES (all hotel roles + staff). manage_team and manage_users are not adminOnly, and capability_overrides ships empty (documented everyone-everything default), so in the out-of-the-box state at every hotel a housekeeper/front-desk/maintenance user passes the capability gate on all team-management routes. Worse, /api/auth/team PUT has no privilege-escalation matrix (unlike /api/settings/users), so a single line-staff user can reset any non-admin coworker's password (GM/owner account takeover) and promote a coworker to GM/owner.
- **fix:** Give CAPABILITY_META a defaultRoles field; default manage_team/manage_users (and the team_settings cluster) to ['admin','owner','general_manager'] instead of DEFAULT_GRANTED_ROLES. Keep verifyTeamManager's legacy canManageTeam() (admin/owner/GM) as a hard floor that always applies regardless of overrides. Add the same denyRoleChange privilege-escalation matrix that /api/settings/users has to /api/auth/team PUT so password resets and role changes can't be performed by line staff.

## 2. [HIGH] Cross-tenant capability bypass via fleet-wide fan-out paths (org-wide announce + checklist copy-to-properties skip per-target capability check)
- **rec:** fix-with-care  | **cat:** Multi-tenant authorization bypass
- **files:** src/app/api/comms/announce/route.ts, src/app/api/settings/checklists/copy/route.ts
- **root cause:** Both routes check the actor's capability only against their active/source property, then act on a set of target properties authorized purely by raw property_access membership (listAccessiblePropertyIds / canManageHotel) without re-running canForProperty for each target. A non-admin multi-property manager whose post_announcements or manage_checklists was revoked at hotel B via the Access tab can still force a mandatory-read announcement into B (announce orgWide:true) or overwrite B's cleaning/inspection checklists (copy), defeating the exact per-hotel restriction the Access tab exists to enforce.
- **fix:** Filter targets by the per-property capability before acting: await Promise.all(targets.map(p => canForProperty({role}, cap, p))), drop or 400 on disallowed targets, reusing the same override map honored on the single-property path. Apply to both routes (post_announcements in announce, manage_checklists in checklist copy).

## 3. [HIGH] Cross-tenant inventory corruption — order-create stamps any inventory row by client-supplied itemId with no property scope (service-role bypasses RLS)
- **rec:** auto-fix-safe  | **cat:** Multi-tenant data corruption
- **files:** src/lib/ordering/db.ts, src/app/api/inventory/orders/create/route.ts
- **root cause:** createPurchaseOrders passes each cart line's client-supplied itemId straight to stampOrderedItems, which runs supabaseAdmin.from('inventory').update(stamp).eq('id', l.itemId) with no .eq('property_id', pid). supabaseAdmin bypasses RLS, so a manager at hotel A can submit a cart line carrying hotel B's inventory UUID and silently overwrite B's vendor_name, unit_cost, and last_ordered_at. The sibling anon-client updateInventoryItem already adds the property_id guard; this service-role path omits it.
- **fix:** Thread pid into stampOrderedItems and add .eq('property_id', pid) to the update, OR fetch the set of inventory ids for pid once in createPurchaseOrders and drop any cart line whose itemId is not in that set (mirroring vendor resolution). The receive path is safe (its itemIds come from a property-scoped PO) and does not need changing.

## 4. [HIGH] Compliance reminders & life-safety escalations fire at wrong local hour fleet-wide — cron ignores per-property timezone, hardcodes America/Chicago
- **rec:** auto-fix-safe  | **cat:** Timezone correctness / SMS misfire
- **files:** src/app/api/cron/compliance-reminders/route.ts, src/lib/compliance/reminders.ts
- **root cause:** The cron calls runComplianceRemindersForProperty(id) with no timezone, so it defaults to APP_TIMEZONE ('America/Chicago') and gates engineer reminders (10/14) and the GM overdue-escalation (16) on Chicago local time for every property. Non-Central hotels get their nudges and life-safety escalation SMS at the wrong local hour (e.g. Eastern 11am/3pm/5pm, Pacific 8am/12pm/1pm). The lib already accepts and correctly uses a tz arg; the caller just never passes it. This is the established correct pattern in run-daily-report, which the cron deviates from.
- **fix:** Select id, timezone for candidate properties (as run-daily-report does) and pass each property's real timezone: runComplianceRemindersForProperty(p.id, new Date(), p.timezone ?? 'America/Chicago').

## 5. [HIGH] Serial per-property cron fan-out blows the 60s Vercel cap at scale — rules-engine (5-min) and auto-assign (15-min) starve the same tail hotels every tick
- **rec:** needs-reeyen-decision  | **cat:** Scale / silent functional failure
- **files:** src/lib/rules-engine/engine.ts, src/app/api/cron/run-rules-engine/route.ts, src/lib/auto-assign-runner.ts, src/app/api/cron/run-auto-assign/route.ts
- **root cause:** Both crons load every property (no active/PMS filter) and process them in a stable-order serial for-await loop with ~5+ DB round-trips per property and no concurrency, sharding, or wall-clock budget. At fleet scale the loop exceeds maxDuration=60; Vercel kills the function mid-loop and, because order is stable, the same tail hotels are starved every tick — rules-engine never generates their cleaning_tasks (housekeepers see no auto room list), auto-assign never assigns rooms. The sibling run-daily-report already has the deadline-guard pattern these lack. (auto-assign is lower-likelihood: per-property work usually early-returns on an indexed empty read and the engine is idempotent/re-run-each-tick, so it lags rather than permanently starves — but the structural fix is identical.)
- **fix:** Reuse runWithConcurrency from src/lib/parallel.ts (cap ~5) for the per-property fan-out, add a ~50s wall-clock budget that stops claiming new properties and returns a 'degraded' heartbeat, filter to active/PMS-connected properties, and rotate/order by least-recently-evaluated so no hotel is permanently starved. Same fix shape for both crons; add shard support (applyShardFilter) as the fleet outgrows one run.

## 6. [HIGH] Offline-replay idempotency claim never released on write failure — housekeeper actions silently lost on retry and reported as success
- **rec:** fix-with-care  | **cat:** Idempotency / data loss
- **files:** src/app/api/housekeeper/structured-issue/route.ts, src/app/api/housekeeper/add-note/route.ts, src/app/api/housekeeper/mark-for-inspection/route.ts
- **root cause:** These three offline-replayable routes insert the actionId claim row before the real side effect but never delete it on an error/500 path. When the offline queue retries the same actionId after a transient DB/RPC failure, the claim INSERT conflicts, the route takes the dedup branch, and returns ok({deduped:true}) HTTP 200 with the empty payload — so the work order / room note / inspection mark is permanently dropped while the phone shows it as sent. report-found-item already has the correct releaseClaim() pattern on every error path; these three omit it.
- **fix:** Replicate report-found-item's releaseClaim(): delete the claim row on every error/return-500 path and in the outer catch, OR defer the claim INSERT until after the side effect succeeds, OR store a status on the claim and only treat a 'succeeded' claim as a real dedup hit so failed/empty claims are re-attempted.

## 7. [MEDIUM] Front-desk Rush posts housekeeper SMS to a non-existent /api/sms-send route — page silently never sends while UI says 'Housekeeper notified'
- **rec:** fix-with-care  | **cat:** Silent functional failure
- **files:** src/app/api/front-desk/rush/route.ts, src/app/front-desk/_components/RushButton.tsx
- **root cause:** The rush route fans out the notification to ${origin}/api/sms-send, which does not exist (404). The 404 is swallowed (fetch resolves rather than throws), and smsSent is set optimistically before the fetch runs, so the client always shows 'Housekeeper notified'. The is_rush flag and task priority still persist, but the proactive SMS page — the whole point of Rush — never fires, and it fails silently so it reads as housekeepers ignoring rushes.
- **fix:** Replace the fetch-to-/api/sms-send with enqueueSms({propertyId, toPhone, body, idempotencyKey: rush:${pid}:${roomNumber}:${dayBucket}, metadata}) per assigned staff (the same pattern lost-and-found/packages notify-guest already use), and set smsSent:true only after enqueue succeeds rather than optimistically.

## 8. [MEDIUM] Compliance auto-act work orders not idempotent — retry / double-submit / two-phone creates duplicate urgent WOs
- **rec:** needs-reeyen-decision  | **cat:** Race condition / data integrity
- **files:** src/lib/compliance/autoact.ts, src/lib/compliance/store.ts, src/app/api/engineer/log-reading/route.ts, src/app/api/compliance/log-reading/route.ts
- **root cause:** createComplianceWorkOrder does a plain INSERT with no dedupe key, called on every out-of-range reading and failed PM check. The SMS side is deduped per-day, but the WO is not, and the manual reading paths send no idempotencyKey so the reading insert isn't deduped either. A network retry, remount re-fire, or two devices logging the same reading produces multiple urgent WOs for the same event, inflating WO counts and repair-spend rollups.
- **fix:** Derive a stable dedupe key (compliance:oor:${pid}:${typeId}:${todayStr(tz)}) for the auto-act WO, backed by a partial-unique index on work_orders(property_id, source, dedupe_key) with upsert (or a pre-insert existence check for an open compliance WO for that type+day). Pass an idempotencyKey from the manual reading save paths so a retried submit collapses to one reading + one WO.

## 9. [MEDIUM] CapEx decision route has no current-status guard — completed/in-progress capital project can be reset, nulling approved_at and wiping the approver audit
- **rec:** auto-fix-safe  | **cat:** State machine / financial data corruption
- **files:** src/lib/financials/db.ts, src/app/api/financials/capex/decision/route.ts
- **root cause:** decideCapex unconditionally overwrites status/decided_at/decision_notes/approved_by (and nulls approved_at for reject/revisions) on any matching row with no status check, while the sibling progress route correctly guards the transition. A stale second tab re-clicking Approve/Reject, or any retried POST, can regress a project that already has logged line-item spend back to rejected/approved, destroying the approval audit trail the CapEx binder is meant to make un-bypassable.
- **fix:** Add a status predicate to the UPDATE — .in('status', ['requested','revisions_needed']) — so a stale/forged decision matches 0 rows and returns 409 invalid_transition. Prefer the in-DB predicate (also closes the concurrent-decider race) over an application-level pre-check.

## 10. [MEDIUM] Receiving a PO loses stock on partial DB failure — qty_received committed before the stock increment, stock-write error swallowed, non-retryable
- **rec:** needs-reeyen-decision  | **cat:** Data corruption
- **files:** src/lib/ordering/db.ts, src/app/api/inventory/orders/receive/route.ts
- **root cause:** receivePurchaseOrder advances qty_received (the idempotency anchor) first, then applies stock deltas in a separate non-transactional batch whose error is only logged, never thrown. If the inventory.current_stock write fails, the PO still flips to received and returns ok:true, but stock was never incremented — and because qty_received is already at target, a re-submit computes delta<=0 and skips, so the operator cannot retry. The ledger insert is also fire-and-forget, compounding divergence.
- **fix:** Throw on the stock-update error so the route 500s and the operator retries, and bump qty_received only AFTER the stock delta succeeds. Wrap receiving in a single RPC/transaction if feasible; at minimum surface per-item write failures in the response (like shortLines) and stop swallowing the ledger-insert error.

## 11. [MEDIUM] accept-invite consumes the invite non-atomically (no compare-and-swap) — concurrent double-submit can double-redeem and delete the legitimate acceptor's login
- **rec:** fix-with-care  | **cat:** Race condition / invite abuse
- **files:** src/app/api/auth/accept-invite/route.ts, src/lib/auth/auth-create-user.ts
- **root cause:** The invite is marked consumed last (update({accepted_at}).eq('id', invite.id) with no .is('accepted_at', null) CAS), after createOrReclaimAuthUser and the accounts INSERT. Two POSTs with the same valid token both pass the read-time null check. With no unique constraint on accounts.data_user_id, a same-email race can drive createOrReclaimAuthUser into the orphan-reclaim path and DELETE the first acceptor's freshly-created auth login (ON DELETE CASCADE wipes their accounts row). The sibling housekeeper exchange-code already claims atomically with .is('consumed_at', null).
- **fix:** Claim the invite atomically up front, before any side effect: update({accepted_at}).eq('id', invite.id).is('accepted_at', null).select('id').maybeSingle(); 410 if null. Roll the claim back (accepted_at=null) on a later auth/accounts failure, mirroring releaseClaim.

## 12. [MEDIUM] Single-hotel manager can reset password / deactivate a multi-hotel staff account — global credential mutation locks the user out of (or hijacks) other hotels
- **rec:** needs-reeyen-decision  | **cat:** Multi-tenant authorization
- **files:** src/app/api/auth/team/route.ts, src/app/api/settings/users/route.ts
- **root cause:** PUT /api/auth/team and /api/settings/users deactivate only verify the target has access to THIS hotel, then mutate the GLOBAL auth.users credential (password reset / ban_duration). Because one Supabase credential is shared across all hotels in an account's property_access, a manager at one shared hotel can reset the password (gaining a working credential) or ban the sign-in of a user who also belongs to other hotels they have no authority over. Bounded to the admin-provisioned multi-hotel population (regional staff, multi-property owners).
- **fix:** Before any global credential mutation, require the caller can manage EVERY hotel in target.property_access, or block when the target belongs to hotels outside the caller's scope ('only an admin can reset this user — they belong to other hotels'). For deactivate on a multi-hotel target, prefer detaching this hotelId from property_access over a global auth ban.

## 13. [MEDIUM] Scheduled-reports cron drops deferred reports for the day (exact-hour due-check + no catch-up) AND writes a false-green heartbeat when individual schedules error
- **rec:** needs-reeyen-decision  | **cat:** Scale / partial-failure monitoring blind spot
- **files:** src/app/api/cron/run-scheduled-reports/route.ts
- **root cause:** Two compounding defects in one cron. (1) When the 55s budget is hit, remaining schedules are marked 'deferred' and continued without markScheduleRun, so last_run_date is never set; because isDue requires parts.hour === s.hourLocal exactly and the cron runs hourly, the next run no longer matches and the schedule is silently dropped for the day with no error and no 'sent' marker. Offload the AI summary / email fan-out so one slow hotel can't starve the budget. (2) Compute anyError = results.some(r => r.status === 'error') and pass status: anyError ? 'degraded' : 'ok' to writeCronHeartbeat, matching every other cron.

## 14. [MEDIUM] Daily/weekly report cron strands a hotel's report on timeout — placeholder-first idempotency converts a mid-flight kill into a permanent 'already sent' skip
- **rec:** needs-reeyen-decision  | **cat:** Partial-failure / scale
- **files:** src/app/api/cron/run-daily-report/route.ts, src/app/api/cron/run-weekly-report/route.ts
- **root cause:** Both crons iterate properties serially with no outer wall-clock budget (only a per-recipient 45s deadline inside one property), insert the report_runs placeholder FIRST as the idempotency guard, then build the payload (4s ML + read fan-out) and send emails. If Vercel kills the function (maxDuration=60) after a property's placeholder lands but before its sends complete, the next tick sees the row and returns 'skipped_already_sent', so that hotel gets no report that day with no error surfaced. Mirror into run-weekly-report.

## 16. [MEDIUM] Stripe webhook can write subscription_status='trial' without trial_ends_at — CHECK violation re-throws and the event 500-retries for 3 days, stranding billing sync
- **rec:** fix-with-care  | **cat:** Crashes / billing correctness
- **files:** src/app/api/stripe/webhook/route.ts
- **root cause:** On customer.subscription.updated with status 'trialing', mapStripeStatus returns 'trial' and the handler patches subscription_status='trial' without setting trial_ends_at. Because converted rows have trial_ends_at=NULL, this violates the properties_trial_has_end CHECK (23514). The handler treats every 23514 as 'migration 0038 not applied' and re-throws; the outer catch deletes the dedupe row and returns 500, so Stripe retries the same event for up to 3 days, each retry 500ing, and the property's billing state never updates. Triggered by merchant-side trial grants/extends, subscription-schedule phases, or coupon-driven trials.
- **fix:** When writing subscription_status='trial', also populate trial_ends_at from the Stripe-authoritative value: trial_ends_at: sub.trial_end ? new Date(sub.trial_end*1000).toISOString() : trialEndsAt().toISOString(). Optionally, in the 23514 branch, distinguish the trial-invariant violation from the migration-0038 case and self-heal rather than blanket re-throwing.

## 17. [MEDIUM] Stripe webhook error-path dedupe DELETE is unchecked — a failed delete permanently masks an unprocessed event as 'already processed'
- **rec:** fix-with-care  | **cat:** Race condition / missing idempotency
- **files:** src/app/api/stripe/webhook/route.ts
- **root cause:** After a handler throws, the code deletes the dedupe row so Stripe's retry can reprocess, but the awaited DELETE's error is never checked (supabase-js returns errors in-band, doesn't throw). If the DELETE fails, the row survives; the next retry's INSERT hits 23505 and the webhook returns 200 'deduped', marking the event processed though the property update never ran. A checkout.session.completed lost this way leaves a paid customer stuck in trial/past_due with no further Stripe retries and no reconciliation path.
- **fix:** Inspect the DELETE result; on error, log loudly and return 500 so Stripe retries. More robustly, make dedupe two-phase ('claimed' on insert, 'completed' only after the handler succeeds) and treat claimed-but-not-completed rows as reprocessable, so a lost delete can't permanently mask an unprocessed event.

## 18. [MEDIUM] apply-to-all can wipe every hotel's capability restrictions when the source hotel has none / is invalid — no source-exists guard, no atomicity, no audit log
- **rec:** fix-with-care  | **cat:** Data corruption
- **files:** src/app/api/admin/access/apply-to-all/route.ts
- **root cause:** The route validates propertyId is a UUID but never confirms it exists, reads the source's capability_overrides (empty for a non-existent UUID or a hotel with no restrictions — the default state of every unconfigured hotel), then loops deleting every other hotel's overrides and only inserts when rows.length>0. So 'Apply to all' from an unconfigured/default source silently resets the entire fleet's access matrix to everyone-everything, with no transaction (continue-on-error strands hotels half-applied) and no admin_audit_log entry.

(3) Write an admin_audit_log entry with source + hotelsUpdated. (4) Do the per-hotel delete+insert inside a single RPC/transaction.

## 19. [MEDIUM] Admin monitoring/analytics panels truncate or undercount at fleet scale via fixed row caps (.limit) and client-side bucketing
- **rec:** needs-reeyen-decision  | **cat:** Scale / admin blind spot
- **files:** src/app/api/admin/scheduled-jobs/route.ts, src/app/api/admin/activity/route.ts, src/app/api/admin/per-hotel-economics/route.ts

## 20. [MEDIUM] Activity-log PDF export mangles em-dashes and non-Latin-1 characters — corrupts the audit/compliance paper trail
- **rec:** auto-fix-safe  | **cat:** Data corruption (rendering)
- **files:** src/lib/activity-log/export.ts
- **root cause:** The hand-rolled PDF writer emits the content stream and computes /Length with latin1 ('binary') encoding, and escapePdfString only escapes backslash/parens. Any code point above U+00FF is truncated to one byte. The activity_log DB trigger writes em-dashes (U+2014) into routine English descriptions ('Work order created on Room 204 — Leaky faucet', 'role changed from … — …'), so common audit rows render with garbage/control characters for the current EN customer — not just hypothetical CJK names. CSV/XLSX exports are UTF-8 and unaffected.
- **fix:** In escapePdfString, map any char with codePointAt > 255 to a placeholder ('?') so output is always valid Latin-1 and renders predictably (keeps /Length self-consistent since it's computed after escaping). Surviving Latin-1 accents (á/é/í/ñ) are untouched. Longer-term, switch to UTF-16BE text strings with a supporting font.

## 24. [LOW] Unbounded Promise.allSettled fan-out in nudges/check and compliance-reminders crons — request-burst stampede at scale (codebase has a bounded helper these skip)
- **rec:** auto-fix-safe  | **cat:** Scale
- **files:** src/app/api/agent/nudges/check/route.ts, src/app/api/cron/compliance-reminders/route.ts
- **root cause:** Both crons fan out with no concurrency cap (Promise.allSettled(propertyIds.map(...))), each per-property worker issuing many PostgREST round-trips. At fleet scale this fires a thundering-herd of HTTP requests against Supabase's REST layer every 5 min / hourly, spiking head-of-line latency for the live app. runWithConcurrency (cap 5) in src/lib/parallel.ts was purpose-built for exactly this and is used by 8+ sibling crons; these two are the outliers.
