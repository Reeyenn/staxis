# Appendix D — First-Hotel Onboarding Readiness

_Day-1 risks + playbook for Comfort Suites Beaumont, sorted by severity._

| Severity | Item | Detail | Action |
|---|---|---|---|
| **blocker** | PMS connection (the CUA login + scrape) is the single biggest thing that can hang in front of the customer | Step 6 of onboarding ('Learning your PMS') runs entirely on the separate Fly robot that logs into Choice Advantage. There's a pre-built Choice Advantage 'recipe' seeded in the system (so it should NOT need an expensive from-scratch learn), but that recipe was copied from the OLD scraper's button locations back in May. If Choice Advantage changed its screens since then, the robot reads zero rooms, tries to self-heal, and the owner can sit on a spinner. If login fails 5 times in an hour the robot gives up silently with no auto-recovery and the app shows empty. | Before the owner ever sees the screen, log the robot into the REAL Comfort Suites Choice Advantage account yourself and confirm rooms, arrivals and departures are actually flowing (not zero). Get the Choice Advantage map showing green/'promoted'. Babysit the robot machine during the live session. Have the login credentials confirmed-working in hand. | **blocker** | If Choice Advantage texts a security code (MFA) during login, YOU must be watching and type it within ~10 minutes | When the robot logs in and the PMS fires an SMS verification code, the robot pauses and waits for a human to type it on the admin screen. | **blocker** | PRE-FLIGHT: Get the Choice Advantage connection green BEFORE the owner ever logs in | This is the single highest-risk step. You must confirm that shared recipe is healthy and actually pulling Comfort Suites' real data days ahead — not discover it's broken on go-live day. | 2-3 days before: in admin, run the recipe pre-build for Comfort Suites (the regenerate-recipe action), then confirm on /admin/property-sessions that the hotel shows 'alive', a recent heartbeat, and that real rooms/arrivals/departures numbers are populating. Do NOT let the owner's session be the first time the live learn runs. If feeds are empty, fix it now while there's no audience. |
| **blocker** | PRE-FLIGHT: Have the PMS login + MFA ready, and be ready to type the MFA code within ~10 minutes | Choice Advantage can throw a text-message security code (MFA) during login. If it does, the connection stalls until someone types that code on the live mapping screen — and the window is only about 10 minutes. There's no async hand-off; you have to be watching. Choice also added a mandatory Okta-migration nag screen that previously caused a login spiral (that fix is in the live code now, but verify it holds). | Have the exact Comfort Suites Choice Advantage username/password confirmed working by logging in manually yourself first. Know which phone receives the MFA text and have it in hand. Pre-click 'trust this device' so MFA stops firing on future logins. Watch /admin/property-sessions during the first connect so you can resolve any MFA prompt immediately via /admin/mfa-resume. |
| **blocker** | PRE-FLIGHT: Always work from fresh live code, never this Mac's local copy | This machine's working copy is 298 commits behind the live site and shows core files as deleted. Anything built, tested, or 'verified' against this local copy is testing the wrong product and could ship a regression to the live hotel during onboarding week (it would clobber recent fixes to sign-in, 2FA, and dashboard honesty). | Do not build/deploy from the Desktop checkout. Use a fresh checkout of the live code for any change between now and go-live. If any fix is needed, make it on a fresh branch off the live code and only ship when green. |
| **high** | There is NO 'connect my PMS later' escape — a stuck PMS step blocks the whole onboarding | The wizard has no way to skip the PMS connection and finish onboarding. If the credentials aren't handy or the learn keeps failing on the day, the owner is stuck on that step with only a 5-minute 'something's wrong' card. The whole flow funnels through the riskiest step with no off-ramp. | Since you can't add a skip in a few days, mitigate operationally: have the PMS already connected and green BEFORE the owner starts, so they breeze past Step 6 in seconds. Treat 'PMS pre-connected' as a hard prerequisite for booking the onboarding session. |
| **high** | 'All set' can show even when the PMS isn't actually live yet | If the robot only partially learns the PMS (an outcome called 'park_draft'/'park_partial'), the owner is still advanced toward the 'All set' completion screen — but live PMS data does NOT flow until you click 'Promote' in admin. The owner can finish believing they're fully connected when feeds are still off. | After onboarding, verify in admin that the map is 'promoted' and feeds are flowing, not parked. If it parked, promote it and confirm rooms/arrivals appear before telling the owner they're live. Don't rely on the 'All set' screen as proof of connection. |
| **high** | The dashboard, ROI page, and 'savings' story will look empty for the first weeks — by design | The main dashboard's revenue/labor numbers and charts are intentionally shown ONLY on demo hotels; a real hotel correctly sees 'no history yet.' The ROI/'you saved $X' page needs ~90 days of daily logs. So on day 1 the polished demo and the hotel's real (sparse) account look very different. If you demo the demo hotel as if it's theirs, the gap is jarring. | Don't demo synthetic dashboards as the hotel's own data. Frame day 1 honestly: 'this fills in as your team uses it.' Lead the demo with what IS real on day 1 — live room status and arrivals/departures from the PMS, and the housekeeper phone flow — not the financial charts. |
| **high** | The labor-saving AI (headcount recommendation) is currently non-functional against the live database | The machine-learning service that predicts 'you need 5 housekeepers tomorrow' reads a database table (plan_snapshots) that was DELETED when the new PMS robot replaced the old scraper. So demand predictions return 'no data' every run. On top of that, the schedule that's supposed to run training/predictions automatically is fully turned OFF (disabled because it kept failing and emailing errors). Confirmed in the live code. | Do NOT pitch or demo the 'AI tells you tomorrow's staffing' feature at this onboarding — it will look broken or stuck on a generic estimate. Either keep it off the table for now, or schedule the fix (repoint ML to the daily_logs table + re-enable the schedule) as a separate task before relying on it. Set expectations that staffing AI 'starts learning' over the first weeks. |
| **high** | The AI assistant will confidently report wrong numbers for live housekeeping state | When asked things like 'how many rooms are in progress / on Do-Not-Disturb / flagged with issues / requesting help,' the assistant returns hardcoded zeros because the underlying data table was never built (confirmed: ~12 'TODO' zeros in the live agent code). During a live walkthrough this reads as 'the AI is broken' or gives the owner false operational answers. | Steer any live 'Ask Staxis' demo toward questions it answers correctly (occupancy, room status, who's assigned, inventory, knowledge-hub lookups). Avoid asking it for in-progress / DND / issues / help-requested counts in front of the customer until the housekeeper-state table is built. |
| **high** | Do not build, test, or deploy anything from this Mac's local copy this week | The local checkout on this machine is 298 commits behind the live site and shows core files as deleted/half-edited. Building or pushing from here would ship stale code and could regress live fixes (2FA, sign-in screens, dashboard honesty) right during onboarding week. | For any pre-onboarding fix, work only from a fresh copy of the live code (worktree off origin/main), confirm tests pass, and push to the live branch. Never deploy from the stale Desktop tree. This is a process guardrail, not a feature — but it's the easiest way to accidentally break the live hotel app. |
| **high** | PRE-FLIGHT: Spot-check that Comfort Suites room numbers won't be silently dropped | The data pipeline only accepts room numbers that look like 3-4 digits with an optional trailing letter (e.g. 214, 1203, 214A). Any room labeled differently (a suite code, a letter prefix like 'P1', a name) gets silently thrown away — meaning those rooms just never appear in the app, with no error. For a 74-room Comfort Suites this is probably fine, but a single oddly-numbered suite would vanish. | Pull the real room list from Choice Advantage and eyeball every room number against the 3-4-digit-plus-optional-letter format. If any room breaks the pattern, flag it before go-live so the validator can be loosened for that hotel rather than discovering empty rooms later. |
| **high** | DECISION: Onboard as READ-ONLY and say so plainly — Staxis cannot write back to the PMS yet | Staxis can read everything from Choice Advantage but cannot push anything back into it. The write-back machinery is fully built but the actual click-instructions for the room-status screen were never filled in (the code refuses to run until they are). So any promise like 'Staxis updates your PMS automatically' is currently unfulfillable. Marking a room clean in Staxis will NOT mark it clean in Choice Advantage. | Frame Staxis to the GM as the operations layer on TOP of the PMS, not a replacement that writes into it. Set the expectation explicitly: 'Staxis reads your PMS live and runs your housekeeping/maintenance/inventory; for now your front desk still updates room status in Choice Advantage.' This avoids the trust-killing moment of the customer expecting two-way sync that doesn't exist. |
| **high** | DECISION: Don't demo the /feed home or dashboard money charts as if they're live | The flagship /feed 'decision inbox' home screen is built on sample/mock cards — it is not wired to real data yet. The dashboard's revenue/labor KPIs and charts are synthetic and only show on demo properties; a real new hotel correctly sees 'no history yet'. If you demo the polished demo property and then open the customer's real (empty) account, the gap looks jarring and underwhelming. | Pick the surfaces that are real on day 1 (live room status board, housekeeper SMS page, arrivals/departures occupancy from the PMS, inventory) and demo those on the customer's actual account. If you show /feed or the demo dashboard for inspiration, say plainly 'this view fills in as your data builds up over the first weeks.' |
| **high** | DECISION: Decide whether ML staffing recommendations are shown at all on day 1 — they are currently broken | The AI headcount/labor predictions are effectively non-functional right now: the prediction code still reads a database table that was deleted, so housekeeping demand inference returns 'no data' every run and supply predictions default wrongly. On top of that, the automatic schedule that would run training/prediction is turned off (manual-only). So the 'recommended X housekeepers tomorrow' pill will be blank or stuck on a generic industry estimate, and nothing updates by itself. | Either (a) hide/soft-label the ML staffing recommendation for go-live so the customer doesn't see a broken/empty 'AI', OR (b) fix it first: repoint the ML reads from the deleted table to the daily_logs table (same columns, already populated), then re-enable the ML schedule. If shown without the fix, it looks broken. Inventory 'days until out' survives the break and is safe to show. |
| **high** | GUARDRAIL: Make sure the housekeeper SMS flow actually gets adopted — the entire labor dataset depends on it | The flagship 'AI learns your hotel' value depends on housekeepers tapping Start/Done on their phones. Also note: the AI assistant currently reports rooms-in-progress, DND, issues-flagged, and help-requested as zeros (a backing table was never built), so if you ask the AI 'how many rooms in progress' during a live demo it will confidently say 0. | On go-live day, physically walk one or two housekeepers through opening their SMS link and tapping Start/Done on a real room, in their language (EN/ES supported, plus ht/tl/vi). | **high** | GUARDRAIL: Confirm the connection won't silently die with no alert | Each hotel runs on a single machine with no backup. The health watchdog runs every 5 minutes but a dead 'gave up' state needs a human to notice. | For the first 48 hours, check /admin/property-sessions and /admin/doctor at least twice a day. | **medium** | Staff who TEXT the hotel number expecting a reply just get a generic 'got your message' — looks broken | Inbound texts only do one thing: switch language (ESPAÑOL/ENGLISH).
